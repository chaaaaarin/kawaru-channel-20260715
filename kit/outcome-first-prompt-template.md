# 🎁 Outcome-Firstプロンプトテンプレート

手順を細かく書く「手順書型」から、成果と完了条件だけを書く「Outcome-First型」に切り替えるためのシステムプロンプトの型です。トークン削減とコスト削減の両方に効きます。

**受け取り方**: 下のコードブロック右上のコピーボタンでコピーするか、このファイルをそのままダウンロードしてください。

**使い方**: 各項目の〔　〕を自分のタスクに合わせて書き換えてから、システムプロンプトとして使ってください。既存の長いプロンプトを見直す場合は、まず今のプロンプトをこの型に当てはめ直し、当てはまらない指示（重複ルール・モデルが既にこなせる手順・振る舞いを変えない例示）を削ってください。

```text
【Outcome-Firstプロンプトの型】

# 成果定義
このタスクで達成すべきことは〔具体的な成果物・状態〕です。

# 完了条件（ここまでやれば終わり、という基準）
- 〔条件1：何を満たせば完了と言えるか〕
- 〔条件2〕
- 〔条件3〕

# 制約・安全要件（ALWAYS / NEVERは、ここだけに限定する）
- ALWAYS: 〔本当に外せない不変条件のみ。安全要件・権限の制約など〕
- NEVER: 〔本当に外せない禁止事項のみ〕

# 自律性の境界（承認が必要な操作を1箇所で定義する）
- 次の操作は、実行前に必ずユーザーに確認する: 〔外部への書き込み・破壊的操作・不可逆な操作など〕
- 上記以外の安全な操作（調査・下書き作成・ローカルでの試行など）は、都度の確認なしで進めてよい

# 経路はモデルに委ねる
上記の成果・完了条件・制約を満たす範囲で、具体的な手順・進め方はあなたの判断で決めてよい。
やり方を逐一指示しない。

---
チェック観点（プロンプトを書いた後に、この型に沿っているか自己点検する）:
- 重複しているルールはないか
- モデルが既にこなせる手順まで書いていないか
- 振る舞いを変えない不要な例示を入れていないか
- 指示同士が矛盾していないか（矛盾は指示の欠落より不安定さを生む）
```

**ポイント**: 出力はレビューしてから確定する。特に自律性の境界（承認が必要な操作）は、書き込み・削除・送信など取り返しのつかない操作を必ず含めること。
