# 🎁 モデル選定チェックリスト（Sol / Terra / Luna × エフォート × 速度）

GPT-5.6のSol・Terra・Lunaと推論エフォート（努力設定）・速度モードを、タスクの難易度に応じてどう組み合わせるかの早見チェックリストです。

**受け取り方**: 下のコードブロック右上のコピーボタンでコピーするか、このファイルをそのままダウンロードしてください。

**使い方**: タスクを始める前にこのチェックリストを上から確認し、最初に条件が当てはまったモデル・エフォートから試してください。

```text
【GPT-5.6 モデル選定チェックリスト】

■ 原則：計画は賢いモデル、実行は安いモデル
□ 設計判断・仕様決めのような「賢さ」が要る部分だけ上位モデル（Sol High等）に任せる
□ 実装・反復・大量処理のような「量」をこなす部分は下位モデル（Luna等）に渡す
  （迷ったらこの原則に立ち返ると判断がぶれない）

■ ステップ0：そもそもワーク/Codexを開く必要があるか？
□ 定義の確認・ちょっとした言い換えなど「簡単な質問」か？
  → YES: ワーク/Codexを開かず、通常のチャットで済ませる（一番安い）

■ ステップ1：タスクの性質を確認する
□ 大量データの一括処理・単純な分類・速度重視のタスクか？
  → YES: Luna（高エフォート設定）から試す
□ 日常業務（下書き作成・要約・一般的なコーディング）か？
  → YES: Terra（標準エフォート）から試す
□ 複雑な意思決定・高価値な成果物・長時間の自律タスクか？
  → YES: Sol を検討する

■ ステップ2：グレードを上げる前に、エフォートを上げてみる
□ Terraで力不足を感じたら、まず「Terra Ultra」を試す
  （Sol Extra Highを検討する前に、Terra Ultraで足りないか確認する）
□ Lunaで力不足を感じたら、まず「Luna 高エフォート」を試す
  （Sol Highより下のグレードを検討する前に、高エフォートのLunaで足りないか確認する）

■ ステップ3：Solを使う場合の判断
□ Sol Ultra（マルチエージェント・最上位設定）は、コスト増に見合う価値があるかを個別に確認する
  （Sol Ultraの追加コストは、Sol Maxほどの価値がないことが多いという指摘がある）

■ ステップ4：試算のダブルチェック
□ 「モデルを1段階下げてエフォートを上げる」場合と
  「モデルはそのままでエフォートも標準」の場合を、
  実際のタスクで両方試算してから決める
  （単純な単価計算だけでは、どちらが安いか分からないことがある）

■ ステップ5：精度が必要なタスクは例外扱いにする
□ 実世界のコード修正・重要な意思決定など「間違えられない」タスクは
  コスパ最優先の判断から外し、実績のあるモデル・グレードを使う

■ ステップ6：速度を上げたいときはFast modeより先にEffortを見直す
□ 応答が遅いと感じたら、まずEffortを1段階下げられないか検討する
  （Fast modeはクレジット消費がStandardの2.5倍。GPT-5.6は自走力が高く、
  Fast modeと組み合わせると1メッセージで利用枠を大きく消費することがある）

■ ステップ7：subagentの暴走を事前に防ぐ
□ グローバルのAGENTS.mdに「指示がない限りsubagentを自律生成しない」旨を
  1行加えているか確認する
  （Solは親と同じモデル・エフォートでsubagentを生成する挙動があり、
  対策なしだと利用枠を想定外に消費しやすい）

■ ステップ8：作業前に利用枠のリセット状況を確認する
□ 重い作業に取りかかる前に、利用制限の一時緩和・リセットの
  公式アナウンスが出ていないか確認する
  （リセット直後の枠が多いタイミングに重い作業を寄せると、
  体感のコストパフォーマンスが変わる）
```

**ポイント**: このチェックリストは公式ベンチマーク（Artificial Analysis Coding Agent Index等）をもとにした目安です。タスクの性質によって最適解は変わるため、重要な用途では必ず実タスクで精度を確認してから本番運用してください。
