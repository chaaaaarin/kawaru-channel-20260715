## 1. Sol・Terra・Lunaで何が変わったのか

GPT-5.6は「太陽・地球・月」を意味するSol・Terra・Lunaという3モデルで構成される。✅ Solが最上位、Terraがバランス型、Lunaが最速・最安のモデルで、番号（5.6）は世代を示し、3モデルはそれぞれ独自のペースで進化していく継続的な層という位置づけになっている。✅（OpenAI公式ページ）

料金は100万トークンあたり、Sol: 入力$5／出力$30、Terra: 入力$2.50／出力$15、Luna: 入力$1／出力$6。✅ 注目したいのは**Solの価格が実は前モデルのGPT-5.5と同額**という点で、値下げではなく「同じ価格でより賢く・より少ないトークンで」という方向の進化になっている。🔶（解説動画で複数人が指摘）

公式ベンチマーク「Agents' Last Exam」のコスト対性能チャートでは、GPT-5.6ファミリー（Sol/Terra/Luna）がClaude Opus 4.8・Claude Fable 5・Gemini 3.1 Pro Previewよりも左上（低コスト・高スコア）に位置しており、同じ予算でより高いスコアを狙えることが示されている。✅（OpenAI公式ページのグラフで確認）

## 2. なぜ今なのか（Before/After）

### Before：mini/nano体系

これまでのGPTシリーズは「本体＋mini＋nano」という接尾辞でグレードを分ける方式だった。用途に応じてモデル名を都度調べる必要があり、どれが一番コスパがいいのかが分かりにくかった。🔶

### After：Sol/Terra/Luna体系

GPT-5.6からは「Sol・Terra・Luna」という独立した名前の3層構造になり、**世代（5.6）とグレードが分離**された。✅ これにより「Terraだけ次のバージョンが早く出る」といった非同期の進化も可能になっている。✅（OpenAI公式ページ）

ChatGPT側のアプリ構成も変わり、チャット（質問・検索・壁打ち向け）とワーク（研究・分析・資料作成など仕事向け、旧Codexアプリ機能を統合）に分かれた。🔶 無料版・Goプランのユーザーは、通常のチャット画面では引き続きGPT-5.5 Instantが使われ、**GPT-5.6 Terraが使えるのはChatGPT WorkとCodexの中だけ**という点は誤解しやすいので注意したい。🔶（複数の海外メディアが報じているが、OpenAI公式の一次テキストでは未確認のため要注意）

## 3. 仕組み・根拠：料金体系とキャッシュ

GPT-5.6では、プロンプトキャッシュの仕組みが強化された。✅ キャッシュ書き込みは非キャッシュ入力単価の1.25倍で課金される一方、**キャッシュ読み取りには引き続きキャッシュ入力に対して90%の割引が適用される**。✅（OpenAI公式ページ）明示的なキャッシュブレークポイントのサポートや、最低30分のキャッシュ有効期間など、より予測可能な設計になっている。✅

Batch API（非同期処理）は同期APIと比べて50%のコスト割引があり、24時間以内に処理が完了する。✅（OpenAI公式ドキュメント）評価実行やデータセット処理など、即時応答が不要な作業はここに回すだけでコストが半分になる。

## 4. 節約テクニック①：モデル・Effort・速度の使い分け

一番効果が大きいのは、タスクの難易度に応じてSol/Terra/Lunaを使い分けること。✅ OpenAI公式のモデルガイダンスでも「まず現在のGPT-5.5/5.4の設定から始めて、1段階下のグレードで試す」ことが推奨されている。✅

具体的な指針として、機械学習教育者として知られるSebastian Raschka氏（@rasbt）が公式ベンチマーク「Artificial Analysis Coding Agent Index」のコスト対性能チャートを基に分析を公開している。🔶（個人の分析。データ自体は公式ベンチマーク由来）

- Terra Ultra級の性能が不要なら、常に高エフォート設定のLunaを使う方が同等以上の性能でより安価
- Sol Highより下のグレードは検討せず、高エフォート設定のLunaを使う
- Sol Extra Highは検討せず、Terra Ultraを使う
- Sol Ultraの追加コストは、Maxほどの価値がないことが多い

TerminalBench 2.1のベンチマークでは、Terra（84.3%）がClaude Fable 5（84.3%）と同スコアながら大幅に低コストという結果も出ている。🔶（ベンチマーク集計サイトの報告。個別の一次ページは未特定）モデル（Sol/Terra/Luna）だけでなく、**Effort（推論の強さ）と速度モード**の2軸を掛け合わせて初めて「一番安くて効率がいい設定」にたどり着ける。

**この使い分けをひと言でまとめると「計画は賢いモデル、実行は安いモデル」。** ✅（OpenAI公式モデルガイダンスの推奨とも一致）設計判断・仕様決めのような賢さが要る部分だけSol High等の上位モデルに任せ、実装・反復・大量処理のような量をこなす部分はLuna等の下位モデルに渡す——この原則で考えると、Effort・速度・モデル選びのすべての判断がぶれにくくなる。

### Effortレベルの使い分け

GPT-5.6は各モデルに、Light（API/CLI表記ではLow）・Medium・High・Extra High・Max・Ultraの6段階でEffort（推論の強さ）を設定できる。✅（OpenAI公式ドキュメント）Ultraは「設定」ではなくEffortの最上位段階そのもので、複数エージェントへの自動タスク委任を伴う。✅（OpenAI公式ドキュメント）

| Effort | 想定する使いどころ |
|---|---|
| Light（Low） | 素早く完結する軽いタスク向け。速度優先 ✅ |
| Medium | **標準の初期値。** 速度と推論の深さのバランスを取りたい通常タスクはまずここから ✅ |
| High | より複雑な問題に、深く考える時間を与えたいとき ✅ |
| Extra High | Highよりさらに深い推論が必要な複雑な問題向け ✅ |
| Max | 最も難しい問題向けに、最大限の推論時間を与える ✅ |
| Ultra | 最大推論に加え、複数エージェントへの自動タスク委任を伴う。ChatGPT WorkはPro/Enterprise、CodexはPlus以上で利用可 ✅ |

Ultraは公式には「最大推論＋自動タスク委任」と位置づけられているが、2026年7月時点ではCodex側の実装バグにより、委任先のサブエージェントが必要以上に高いモデル・Effortで大量生成されてしまう不具合が報告されている。GPT-5.6-solを高負荷で使い込んでいる開発者Theo氏（t3.gg）は「バグが直るまでUltraは避ける」ことを勧めている。🔶（個人の実運用レポート）

### アプリの使い分け：簡単な質問はチャットへ

Effort・モデルの前に、そもそも**どのアプリで聞くか**という節約軸もある。2章で触れたとおり、ChatGPTはチャット（質問・検索・壁打ち向け）とワーク（研究・分析・資料作成など仕事向け、Codexを統合）に分かれている。✅ 定義を確認したいだけ・ちょっとした言い換えが欲しいだけ、といった**簡単な質問はワーク/Codexを開かず通常のチャットで済ませる**のが一番安い。🔶 ワーク/Codexは重いタスク・成果物を作るタスクのために温存し、「調べもの」「壁打ち」レベルの質問にSolクラスの利用枠を使わないようにするだけで、体感の消費ペースが変わる。

### 速度：StandardとFast mode

Fast modeはモデルの応答速度を約1.5倍にする代わりに、クレジット消費がStandardの2.5倍になる（GPT-5.6・GPT-5.5とも同倍率、GPT-5.4は2倍）。✅（OpenAI公式ドキュメント）

GPT-5.5までは、モデルが早めに手を止めて追加指示を求めることが多く、Fast modeで速度を補う運用が合っていた。GPT-5.6は自走してタスクを完了させる力が上がった分、1メッセージだけで利用枠の最大15%を消費することがあり、そこにFast modeの2.5倍を掛け合わせると**1メッセージで5時間枠の40%が溶ける**こともある。🔶（Theo氏の実運用レポート。倍率自体は公式ドキュメントで確認済みだが、5.6での消費実感は個人の報告）Theo氏は「しばらくFast modeは使わない」ことを勧めている。🔶

## 5. 節約テクニック②：Outcome-Firstプロンプト

OpenAIは2026年7月、GPT-5.6 Sol向けに新しいプロンプティングガイダンスを公開した。✅ 骨子は「手順を細かく指示するのではなく、望ましい結果（アウトカム）と完了条件を伝えて、経路はモデルに委ねる」というもの。✅

社内評価では、重複ルール・振る舞いを変えない例示・モデルが既にこなせる手順指示を削った構成で、**評価スコアが10〜15%向上し、トークンは41〜66%削減、コストは33〜67%削減**という結果が出ている。✅（OpenAI公式ドキュメントで確認、AI×プロダクト開発の専門家アカウントの要約とも一致）

ポイント:

- 「まず削る」が新常識。重複ルール・挙動を変えない例示・モデルが既にこなせる手順指示を1つずつ削ってevalで確認する
- 指示の欠落より「指示同士の矛盾」の方が不安定さを生むと明言されている
- 手順書から「成果定義」へ：ステップ指定をやめ、ゴールと完了基準を書いて経路はモデルに委ねる
- ALWAYS/NEVERは安全要件など真の不変条件だけに限定する
- 自律性の境界は1箇所で定義：外部書き込みや破壊的操作だけ承認を求め、安全な操作は止めない

## 6. 節約テクニック③：プロンプトキャッシュとBatch API

3章の仕組みを実際の運用に落とし込むと、次の2つがすぐ効く節約テクニックになる。

- **プロンプトキャッシュを前提にした設計**: システムプロンプトやツール定義など変化しない部分を先頭に固定し、ユーザー固有の情報を末尾に置く。プロンプトの先頭が一致していればキャッシュが効き、読み取りが90%引きになる。✅
- **即時応答が不要な処理はBatch APIへ**: 評価・分類・大量データの処理などは、Batch APIに回すだけでコストが半分になる。✅

## 7. 節約テクニック④：AGENTS.mdでsubagentの暴走を防ぐ

GPT-5.6 Solの目玉機能の一つが、サブエージェントを自律的に立ち上げて並列作業させる力の高さ。✅（OpenAI公式ドキュメント）ただし実装にはまだ粗さが残っていて、**Solが生成するサブエージェントは、常に親と同じモデル・同じEffortレベルを引き継ぐ**。🔶（Theo氏の実運用レポート）Effortを上げたままサブエージェントが次々に生まれると、意図せず利用枠を一気に消費することになる。

対策は大きく2つ。🔶（いずれも個人の実運用レポートベース。公式のベストプラクティスとしての明文化は未確認）

- **Effortレベルを下げる**: highまでならサブエージェントが生まれてもそこまで痛くない。low・mediumはさらに安全
- **グローバルのAGENTS.mdに一文加える**: 「指示があったときだけサブエージェントを生成する」という制約を明記しておくと、Solの過剰な自動生成を抑えられる

```text
指示がない限り、サブエージェント（subagent）を自律的に生成しない。
複数のサブエージェントを使う作業は、必ず事前に確認を取ってから始める。
```

（`hide_spawn_agent_metadata`設定を使ったより踏み込んだ制御方法もあるが、2026年7月時点では開発者コミュニティの実験的な運用に留まる ⚠️）

## 8. 節約テクニック⑤：トークン切れに備える引き継ぎ書

5時間・週次の利用枠は必ずいつか切れる。切れるたびに「今どこまでやったか」をゼロから説明し直すのは、それ自体が大きなトークンの無駄になる。🔶

対策は、**タスクが長引きそうだと感じた時点で「引き継ぎ書」を書いておくこと**。目的・完了条件、ここまでの進捗、次にやること、決定済みの事項、避けるべきことを短くまとめておけば、利用枠が切れた瞬間にClaude Codeなど別のAIツールへそのまま貼り付けて、続きから再開できる。🔶（実運用ベースのTips。特定の一次情報には基づかない一般的な工夫）

- GPT-5.6/Codex側で長時間セッションを組む場合、事前に「引き継ぎ書」の型を決めておくと、切れた後の再開コストがほぼゼロになる
- Codex公式PDF「Codex-maxxing for long-running work」が推奨する`workstream.md`（進捗・優先順位）や`done-definition.md`（完了条件）の考え方とも近い。🔶（別資料の分析に基づく類推）
- 引き継ぎ書のテンプレートはプレゼントキット③として配布している（[キットの中身と受け取り方](#kit)章）

## 9. 個人プランの見直し

ここまでは主にAPI/開発者視点の節約術だが、ChatGPTを個人で使っている場合はプラン選びも見直しの余地がある。🔶（プラン価格は複数の解説記事からの二次情報。最新の正確な金額は公式サイトで要確認）

- Free（$0）・Go（$8/月）：通常のチャットはGPT-5.5 Instantのまま。GPT-5.6 Terraが使えるのはChatGPT WorkとCodexの中だけ
- Plus（$20/月）：チャットでGPT-5.6 Solが使える。個人利用の基準ライン
- Pro（$100〜200/月）：Sol Proモード・高い推論設定・毎日大量に使うヘビーユーザー向け

「なんとなくPlus」ではなく、チャットでSolを使う頻度が低いなら、Work/Codexだけで完結する使い方＋Goプランでも十分なケースがある。

## 11. 注意点：安さだけ追うと起きること

- **安いモデルが常に正解とは限らない**: 実世界のGitHub課題解決を測るSWE-Bench Proでは、Claude Fable 5がGPT-5.6 Solを上回るという報告もある。🔶 ベンチマークの種類によって強みが変わるため、「コスパ最優先」で下位モデルに固定するとタスクによっては精度が落ちる。
- **ティア×エフォートは両方の軸で試算する**: モデルティアを1つ下げてエフォート（推論の強さ）を上げる方が、単純な単価計算より安くなることがある。実装時は両方の組み合わせで試算するのが安全。🔶
- **Claude Fable 5の週次利用制限とのバーター**: Fable 5は週の利用量が50%に制限される時期があり、GPT-5.6の登場を受けて制限が緩和・リセットされた形跡がある。🔶（複数の解説動画が同時期に報告）「安いから」ではなく「使える量」で選ぶ視点も持っておきたい。
- **評価タスクは安い設定で済ませていいのか**: わざと質の低い事業案を「自分の自信作」として提示し、モデルがどれだけ客観的に（甘くならず）採点するかを比べる検証がある。🔶 通常チャットは甘め、高い設定（Ultra）はより辛口・客観的という傾向が報告されており、客観的な判断が必要なタスクでは安さだけを優先しない方がいい場合がある。
- **利用制限は変わりやすい。公式の一時緩和情報を定点観測する**: 2026年7月12日、OpenAI Codexのプロダクトリード Tibo Sottiaux氏（@thsottiaux）が、Plus/Business/Proの全プランを対象に「5時間ごとの利用制限」を一時的に撤廃し、利用枠を全ユーザー分リセットしたと発表した（週次上限は継続）。✅（Tibo Sottiaux氏の投稿、Bleeping Computer・Digital Trends等の複数メディアが同時期に報道）この種の制限緩和・リセットは今後も不定期に発表される可能性が高く、公式アカウントを定期的にチェックして「リセット直後で枠が多いタイミング」を狙うと、体感のコストパフォーマンスが変わってくる。

## 12. 今日からやること

読んだ内容を、実際に触る順番に並べ直すとこうなる。

1. **Effortの原則は「計画は賢いモデル、実行は安いモデル」で考える**（4章）——設計判断だけ上位モデルに任せ、実装・反復は下位モデルに渡す
2. **簡単な質問はワーク/Codexを開かずチャットで済ませる**（4章）——調べもの・壁打ちレベルの質問にSolクラスの利用枠を使わない
3. **Effortはmedium/highから始める。Ultraは今は避ける**（4章）——Codex側のサブエージェント過剰生成バグが直るまでは、Ultraの追加コストに見合わないことが多い
4. **Fast modeはいったんオフにする**（4章）——GPT-5.6は自走力が高く、Fast modeと掛け合わせると1メッセージで利用枠を大きく消費することがある
5. **グローバルのAGENTS.mdに「指示なしでsubagentを生成しない」を1行追記する**（7章）——Solの自動サブエージェント生成による意図しない浪費を防ぐ
6. **長引きそうなタスクは早めに「引き継ぎ書」の型を用意する**（8章）——利用枠が切れても、Claude Codeなど別のツールへそのまま貼って再開できる
7. **完全に定型・スケジュール化された作業は、GPT-5.6以外の軽量・旧世代モデルに固定する**——判断や創造性がほぼ要らないタスクにSolクラスを使うのはオーバースペックになりやすい
8. **プロンプトに明確な「ここで一度止めて」ポイントを入れる**（5章のOutcome-Firstプロンプトと合わせて）——GPT-5.6は指示がないとどんどん先まで進んでしまうため、要所でレビューを挟む設計にしておく
9. **利用枠のリセット・制限緩和の公式アナウンスを定期チェックする**（11章）——「いつもと違うタイミングでリセットされていないか」を見ておくと、コスパの良いタイミングで重い作業を寄せられる

これらは、GPT-5.6に限らず「自走力の高いエージェント型モデル」全般に応用できる考え方でもある。別のモデルを使うときも、同じ項目をそのまま当てはめて確認すると良い。

## 14. まとめ

GPT-5.6は「一番上のモデルを使えば安心」という時代の終わりを告げるアップデートでもある。「計画は賢いモデル、実行は安いモデル」を軸に、Sol/Terra/Lunaとアプリの使い分け、Effort・速度モードの調整、AGENTS.mdでのsubagent制御、Outcome-Firstなプロンプト、プロンプトキャッシュとBatch APIの活用、そして利用枠が切れたときの引き継ぎ書——これらを押さえるだけで、体感コストは大きく変わる。一方でベンチマークによって強みが異なる以上、「安さ」だけを絶対的な指標にせず、タスクごとに試算しながら使い分けるのが、今のところ一番誠実な節約の仕方。
